從哲學角度
設計軟體開發

用 Popper 證偽主義取代瀑布式開發。用熵減原則取代功能堆積。用血清素設計取代多巴胺陷阱。這是一份關於「為什麼這樣做」的完整紀錄。

WHY HOW WHAT
01 / Why

為什麼先問問題，不是先寫程式碼

大多數軟體專案失敗，不是因為寫錯程式碼，而是因為解錯問題。設計哲學的核心：讓每個決策都能回溯到同一個根基 ── 使用者在使用工具的過程中，感受到被尊重、被理解。

先問存在目的，再定義功能，最後設計實作。逆向也要通：從實作追溯到使用者痛點，追不通就是設計有問題。

01

熵減原則

熵增是自然狀態 ── 功能越加越多、介面越來越複雜。熵減是主動選擇：每個元素加進來之前問「它的職責是什麼」，職責不清楚就不加。不做什麼和做什麼一樣重要。

02

外化元框架

使用者隨時知道三件事：現在在哪、下一步是什麼、出錯了怎麼辦。做到了，使用者就有掌控感。掌控感帶來的是血清素 ── 穩定、安定、不焦慮。

03

血清素設計

讓人用完安心離開，不靠上癮留人。工具完成任務，使用者繼續生活。不設計「再來一次」的鉤子，不用通知轟炸、進度條綁架、無限等級誘惑。

04

失敗即 Log

失敗不是身份宣判，是調試紀錄。使用者做錯了，介面說「這樣試試」，不說「你錯了」。開發踩坑了，記進 lessons.md，下次不再踩。失敗是學習的原材料。

問題是假說，不是事實

受 Popper 證偽主義啟發：一個問題只有在「能被證明是假的」時，才值得投入資源解決。不找支持自己的證據，找能推翻自己的證據。

步驟 錯誤方式 證偽方式
定義問題 「使用者需要更好的工具」 「使用者平均花 2 小時做 X，願意付 Y 解決」
找證據 問「你覺得這麻煩嗎」 問「上次做這件事花了多久」
判斷 使用者說「好像有用」 找到會推翻假設的條件，逐一排除

熵減 vs 熵增：兩種產品命運

面對任何「要不要加」的決策，問三個問題：它的職責是什麼？不加它會怎樣？加了之後誰來維護？三個都答不清楚 ── 先不加。

熵增思維 熵減思維
功能 加功能讓產品更強 只加解決真實痛點的功能
需求 使用者說要就加 先挖真正的需求
程式碼 越複雜說明越聰明 越簡單越難被破壞
文件 事後補（通常省略） 文件是設計的一部分
依賴 多 = 功能強 多 = 維護成本高

工具配合人，不是人配合工具

大多數工具的邏輯：「工具提供功能，人去學怎麼用。」這裡的邏輯反過來：先問這個人實際在做什麼，再決定工具長什麼樣。

人群 ── 痛點 ── 場景

在「感覺有個好點子」和「確認值得做」之間，走一條可驗證的思考路徑。每一步都必須能說清楚「為什麼」，不能跳步驟。

02 / How

怎麼落地：SDD 工作流

Spec-Driven Development ── 先定義「對的標準」，再去達到它，再驗證達到了。不是蓋房子的瀑布式，是 Popper 證偽主義的開發循環。

01
discuss
需求不清楚時討論，可跳過
02
propose
spec + design + tasks 三件套
03
analyze
四維一致性檢查，0 findings
04
TDD
失敗矩陣 + 紅燈測試
05
apply
派工執行，紅燈變綠燈
06
archive
完成歸檔

需求變更時，不在程式碼裡偏離 spec ── 先 ingest 更新 spec，重新 analyze，再繼續 apply。Spec 是可推翻的假說，不是不能改的設計圖。

spec.md

功能規格

問題陳述、目標、功能範圍、排除範圍、接受標準。每條接受標準都對應一個紅燈測試。

design.md

設計決策

選了什麼、為什麼選、否決了什麼。六個月後回來看，能立刻理解決策脈絡。

tasks.md

實作清單

依 Wave 分組，每個任務標註工具。不同檔案可並行，同檔案必須串行。

TDD：先證明「這會壞」，再去修

Phase 2（紅燈）和 Phase 3（綠燈）不能顛倒。先有實作再寫測試，測試會在不知不覺中配合實作 ── 失去驗證的意義。

Phase 1

Spec 定義「什麼是對的」

Phase 2

寫測試，證明現在是錯的

Phase 3

寫實作，讓紅燈變綠燈

Phase 4

重構，不改行為只改結構

失敗矩陣：在動手前窮舉失敗點

每個可能失敗的情境都有明確的測試名稱和預期結果。確認清單完整後才開始寫實作。

失敗點 紅燈測試 預期結果
未知的指令 unknown-cmd-throws 拋出明確錯誤訊息
重設後狀態不回初始值 store-reset-defaults 所有欄位歸零
正式環境啟用了測試模組 prod-throws-error 拋出環境錯誤
狀態更新後沒有觸發 activate-updates-status 狀態正確轉換

逆向拆解：從市場推回技術

不是「我會什麼技術就做什麼產品」，而是「市場需要什麼，倒推需要哪些技術特性，再選最適合的候選方案」。

  • 市場需求 ── 目標用戶的核心痛點是什麼？
    • 使用場景決定產品形態（離線需求 = 桌面應用、即時協作 = Web）
  • 產品特性 ── 解決痛點需要哪些技術約束？
    • 離線、本地資料、跨平台、高品質輸出、維護可行性
  • 候選評估 ── 列出 2-4 個候選，逐一比對約束
    • 不以「熟悉度」或「流行度」選技術，以「約束匹配度」選
  • 記錄決策 ── 寫進 design.md，含否決理由
    • 六個月後能回溯：為什麼選 A、為什麼不選 B 和 C

選型決策樹（範例）

每個技術選擇都有對應的約束來源，不是偏好。

需要離線使用嗎？
是 ── 桌面應用或 PWA
需要跨平台嗎？
是 ── 排除原生方案
前端生態 + 小 bundle？
是 ── Tauri（Rust + Web）
自己做 auth / billing？
否 ── 用 SaaS boilerplate，省 6-10 週
維護成本可控？
是 ── 差異化功能自己做，標準化用現成

開發環境三模式

消除打包等待是熵減實踐。Mock Backend 讓瀏覽器模式能跑完整流程，迭代速度從 3 分鐘降到 1 秒以內。

pnpm dev

瀏覽器模式

< 1 秒

日常主力。Mock Backend 模擬全部 IPC，熱重載即時回饋。UI 開發、邏輯測試、狀態管理全在這做。

tauri dev

整合模式

~2 分鐘

驗證真實 IPC、本地資料庫、檔案存取。實作新 command 後來這裡確認 Rust 後端正確。

tauri build

打包模式

~3 分鐘

Release 前的最終驗收。完整編譯 + 打包，產出可分發的安裝檔。

03 / What

產出物概覽

哲學和流程最終要落地成可用的東西。以下是這套方法論實際產出的文件和工具。

SPEC

功能規格文件

每個功能的問題陳述、目標、範圍、排除範圍、接受標準。是可推翻的假說，不是凍結的需求。

DESIGN

設計決策紀錄

每個技術選型的完整脈絡：選了什麼、為什麼、否決了什麼、否決的理由。讓接手的人不用猜意圖。

TEST

失敗矩陣 + 紅燈測試

窮舉每個可能失敗的情境，對應測試名稱和預期結果。「完成」的定義：所有紅燈變綠燈。

TASK

分波任務清單

依序分 Wave，每個任務標註工具和負責方。不同檔案並行，同檔案串行。完成即勾選。

LOG

踩坑紀錄

每次失敗即時記進 lessons.md：問題、根因、解法、教訓。同一個坑不踩第二次。

ARCH

逆向拆解樹

從市場需求到技術選型的完整推導鏈。每個選擇都有約束來源，不是偏好。
