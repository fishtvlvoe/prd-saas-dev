# 技術選型決策樹

這個檔案解決的問題：提供 AIRE 技術棧每個選型的決策路徑，以及通用的選型框架，讓未來的決策可以用相同的邏輯走一遍。

---

## AIRE 技術選型決策路徑

### 決策 1：桌面 App vs Web App

```
問題：不動產說明書工具應該是桌面 App 還是 Web App？

約束清單：
  ├── 離線可用（客戶現場不一定有網路）→ 偏向桌面
  ├── 本地檔案輸出（PDF 存到桌面）→ 偏向桌面
  ├── 客戶資料本地存儲（隱私）→ 偏向桌面
  ├── 不需要多設備同步→ 桌面足夠
  └── 買斷制定價（不訂閱）→ 桌面更自然

結論：桌面 App
```

### 決策 2：Tauri vs Electron vs 原生

```
已知：需要桌面 App，支援 macOS + Windows

約束清單：
  ├── 一人公司維護 → 代碼量要可控
  ├── 前端生態熟悉（React）→ 排除 Qt / wxWidgets
  ├── 分發檔案要小（買斷制，使用者直接下載）→ 偏向小 bundle
  ├── 長期維護成本要低 → 偏向活躍社群
  └── 需要系統層級存取（本地 SQLite、檔案 I/O）→ 需要原生後端

候選比較：
  ├── Electron：bundle 80-150MB，Node.js 後端，社群最大，但資源消耗高
  ├── Tauri：bundle ~5MB，Rust 後端，社群成長快，資源消耗低
  └── 原生 Swift / WPF：只能單平台，排除

決策：Tauri
理由：
  - bundle 5MB vs 150MB，使用者體驗差距明顯
  - Rust 後端的安全性優勢（授權驗證不怕被逆向）
  - AIRE 的後端邏輯不複雜（主要是 SQLite CRUD），Rust 入門成本可接受
否決 Electron：資源消耗高，bundle 過大
否決 原生：無法跨平台
```

### 決策 3：Next.js vs Vite + React

```
已知：前端用 Tauri，需要選一個 React 框架

約束清單：
  ├── 需要路由（多頁面 App）→ 需要路由方案
  ├── 未來可能有 SaaS 版（web）→ SSR 能力有用
  ├── PDF 渲染需要 React 生態 → React 為前提
  └── 長期維護 → 選成熟方案

候選比較：
  ├── Next.js 16：完整框架，SSR/SSG/CSR，路由內建，生態豐富
  ├── Vite + React Router：輕量，但要自己組合工具
  └── Remix：更強 SSR，但 Tauri 環境不需要 SSR

決策：Next.js 16
理由：全功能框架，未來若出 web 版本可直接複用，不需要重新整合路由和 SSR
否決 Vite + React Router：組合工具增加維護成本
否決 Remix：過於偏向 SSR，Tauri 環境不適合
```

### 決策 4：SQLite vs PostgreSQL vs 其他

```
已知：需要本地資料庫（桌面 App，離線）

約束清單：
  ├── 離線可用 → 本地資料庫，排除雲端 DB
  ├── 嵌入式（不需要安裝伺服器）→ SQLite
  ├── 一人公司維護 → 避免複雜的 DB 設定
  └── 資料量不大（每個使用者自己的案件，幾百筆）→ 不需要高效能 DB

候選比較：
  ├── SQLite：嵌入式，零設定，Rust 生態完整（rusqlite）
  ├── PostgreSQL：需要伺服器，不適合桌面
  └── IndexedDB：Web 技術，不適合 Rust 後端存取

決策：SQLite（透過 rusqlite）
理由：嵌入式、零設定、Rust 支援成熟、資料量適合
否決 PostgreSQL：桌面 App 不適合需要伺服器的 DB
```

### 決策 5：PDF 渲染方案

```
已知：需要高品質 PDF 輸出，格式要符合法律文件標準

約束清單：
  ├── 需要精確的排版控制（法規條款格式）→ 排除 Print to PDF
  ├── React 生態 → 偏向 React-based 方案
  ├── 支援中文字型 → 關鍵需求
  └── 客製化程度高（logo、主題、格式）→ 要有彈性

候選比較：
  ├── @react-pdf/renderer：React 聲明式 API，支援中文字型，客製化強
  ├── PDFKit（原生 Node.js）：低階 API，要自己管 layout
  ├── Puppeteer（Print to PDF）：格式控制有限，且增加依賴體積
  └── WeasyPrint（Python）：跨語言整合複雜

決策：@react-pdf/renderer
理由：React 聲明式 API 和前端代碼風格一致，中文支援完整，客製化彈性夠
否決 PDFKit：低階 API 要自己管 layout，開發成本高
否決 Puppeteer：格式控制有限，Tauri 環境中 headless Chrome 依賴太重
```

## 通用決策框架

**每次技術選型，走這個流程**：

```
Step 1：列出技術約束（從商業需求推導，不是從技術偏好）
Step 2：確認候選方案（通常 2-4 個）
Step 3：逐一比對候選 vs 約束
Step 4：選出最符合所有約束的方案
Step 5：記錄決策理由 + 否決理由（寫進 design.md）
```

**決策記錄格式（design.md）**：

```markdown
## D[N] — 決策名稱

選擇：[方案名]

理由：
- [約束 1]：[方案名] 滿足，因為...
- [約束 2]：[方案名] 滿足，因為...

否決 [方案 B]：[為什麼不選，哪個約束不滿足]
否決 [方案 C]：[為什麼不選，哪個約束不滿足]
```

## 選型的陷阱

**陷阱 1：技術 FOMO（Fear of Missing Out）**
「這個技術很新，大家都在說，應該選它」→ 新 ≠ 適合

**陷阱 2：過度工程**
「未來可能需要這個功能，所以現在就選能支援的方案」→ 先做當前需要的，等真正需要再重新評估

**陷阱 3：忽略維護成本**
「這個方案功能最強」→ 功能強通常意味著學習曲線和維護成本都高

**陷阱 4：以熟悉度代替適合度**
「我熟悉 X，所以用 X」→ 熟悉 ≠ 適合這個問題

**正確的問法**：
- 「這個技術解決了我哪個具體的約束？」
- 「如果不選這個，最大的代價是什麼？」
- 「三年後，誰來維護這個技術決策？」
