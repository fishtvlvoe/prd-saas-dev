# 逆向拆解法

這個檔案解決的問題：提供一個從「市場需求」反推「技術選型」的可操作框架，讓每個技術決策都能回溯到商業理由。

---

## 核心思路

正向思考（常見的錯誤方式）：
```
我會用 X 技術
    ↓
用 X 能做什麼
    ↓
做一個 X 的應用
```

逆向拆解（正確方式）：
```
市場有什麼需求（使用者有什麼痛點）
    ↓
解決這個需求需要什麼特性（離線？速度？跨平台？）
    ↓
什麼技術能提供這些特性
    ↓
在多個候選技術中，哪個最符合（不只是技術能力，也包含維護成本、學習曲線、社群活躍度）
```

---

## AIRE 的逆向拆解過程

### 第一層：市場定位

```
目標用戶：台灣不動產經紀人（持牌、有自己的案子）
核心痛點：說明書製作耗時（2-3 小時/份）且有合規風險
商業模式：NT$30,000 買斷制（一次性購買，長期使用）
```

### 第二層：產品特性需求

買斷制 + 長期使用 → 需要哪些特性：

| 需求 | 原因 |
|------|------|
| 離線可用 | 經紀人在客戶現場，不一定有穩定網路 |
| 本地資料 | 客戶資料不上雲（隱私考量，買斷制不訂閱） |
| 桌面 App（非網頁） | 需要讀寫本地檔案（PDF 輸出到桌面）；不需要多設備同步 |
| macOS + Windows | 台灣市場兩個平台都有 |
| 高品質 PDF 輸出 | 法律文件，格式要精確 |
| 長期維護可行性 | 一人公司，代碼量要可控 |

### 第三層：技術選型

從需求到技術：

```
離線可用 + 本地資料 + 桌面 App
    → 桌面應用框架（原生 App 或跨平台框架）

macOS + Windows 都要
    → 跨平台框架（排除純 macOS 方案）

長期維護可行性（一人公司）
    → 用前端生態（React/Next.js），不用 Qt/wxWidgets
    → 選有活躍社群的框架

跨平台 + 用前端生態
    → Tauri（Rust 後端 + Web 前端）或 Electron（Node.js 後端 + Web 前端）
```

### 第四層：Tauri vs Electron

| 比較維度 | Tauri | Electron |
|---------|-------|---------|
| Bundle 大小 | ~5MB | ~80-150MB |
| 記憶體使用 | 低（Rust 後端）| 高（Node.js + Chromium）|
| 安全性 | 高（Rust，無 Node.js 漏洞面）| 中 |
| 社群活躍度 | 高（快速成長）| 非常高（成熟）|
| 學習曲線 | Rust 入門成本 | 低（純 JS/Node）|
| 長期維護 | 較低維護成本（小 bundle）| 較高維護成本 |

決策：選 Tauri。理由：
- 買斷制產品需要分發 .app / .msi，5MB vs 150MB 的差距在分發和安裝體驗上很明顯
- AIRE 的 Rust 後端邏輯不複雜（主要是 SQLite CRUD + 授權驗證），Rust 入門成本可以接受
- 長期維護成本較低

### 第五層：PDF 渲染

```
需要高品質 PDF + 法律文件格式精確 + React 生態
    → @react-pdf/renderer（在 React 中聲明式寫 PDF）
    → 不用原生 PDF 庫（PDFKit 等）：要自己管 layout，很痛苦
    → 不用 Print to PDF（瀏覽器 API）：格式控制有限
```

## 逆向拆解的邏輯樹

使用逆向拆解時，先把邏輯樹畫出來：

```
[商業目標]
  ├── [約束 1]（技術邊界）
  │     ├── 候選 A
  │     ├── 候選 B
  │     └── 候選 C → 選這個，原因是...
  ├── [約束 2]
  │     └── ...
  └── [約束 3]
        └── ...
```

每個選擇都有對應的約束（來自商業需求），而不是「我熟悉這個技術」或「這個技術很新很酷」。

---

## 常見的技術選型錯誤

**錯誤 1：因為熟悉就選**
「我熟悉 Vue，所以用 Vue」→ 熟悉 ≠ 適合這個需求

**錯誤 2：因為新穎就選**
「這個框架很新，用新的比較好」→ 新穎 ≠ 適合生產環境

**錯誤 3：沒有考慮維護成本**
「功能強大，所以選這個」→ 強大的工具通常有更高的學習曲線和維護成本

**錯誤 4：沒有考慮生態**
「這個庫非常好用」→ 如果社群不活躍，兩年後你就是在維護一個孤兒依賴

---

## 通用邏輯樹模板

```
商業目標 / 使用者需求
    ↓
技術約束（從需求推導出來的）
    ├── 約束 A → 候選技術...
    ├── 約束 B → 候選技術...
    └── 約束 C → 候選技術...
         ↓
    交集（所有約束都能滿足的）
         ↓
    次要比較（成本、生態、維護）
         ↓
    最終選型 + 決策理由（寫進 design.md）
```

每次做技術選型，走一遍這個流程，然後把決策理由（包含否決方案的原因）記進文件。

六個月後回來看，能立刻理解當初為什麼這樣選。
