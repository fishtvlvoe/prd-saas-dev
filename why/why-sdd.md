# 為什麼先寫 Spec 再寫 Code（SDD）

這個檔案解決的問題：解釋 SDD（Spec-Driven Development）不是多餘的文書工作，而是讓整個開發更快、更少返工的根本原因。

---

## SDD 是 Popper 證偽主義，不是蓋房子

常見的誤解：SDD 就是把設計圖畫好再施工，像蓋房子一樣。

這個比喻錯了。

蓋房子：設計圖確定了就不變，施工是執行，沒有「發現設計錯了就重來」的循環。

SDD：更像實驗物理學。
1. 提出假說（spec）：「這個功能應該這樣運作」
2. 設計實驗（紅燈測試）：「如果假說是對的，這個測試應該會通過」
3. 執行實驗（寫實作）：讓紅燈變綠燈
4. 發現假說需要修正：更新 spec，重新提案（ingest → propose 循環）

**Spec 是可以被推翻的假說，不是不能改的設計圖。**

---

## 傳統開發 vs SDD

| | 傳統開發 | SDD |
|--|---------|-----|
| 開始 | 直接寫 code | 先寫 spec |
| 文件 | 事後補（通常省略）| 文件是設計的一部分 |
| 需求變更 | 代碼改完，不知道為什麼改 | spec 先改，code 跟著改，有完整脈絡 |
| 問「為什麼這樣做」 | 去翻 git blame，猜意圖 | 去翻 spec.md，有明確的決策理由 |
| Code Review | 看 code 猜意圖 | 比對 code 和 spec，看有沒有背離 |
| 重構 | 怕改壞，不敢動 | 有 spec 和測試當安全網，可以大膽重構 |
| 技術債 | 越積越多 | spec 明確了邊界，技術債有地方被看見 |

## Spectra 工作流

Spectra 是 Fish 使用的 SDD 工具框架。完整工作流：

```
discuss（可選）
    ↓
propose（必做）
    — spec.md：定義功能範圍、接受標準
    — design.md：設計決策（選擇了什麼、為什麼、否決了什麼）
    — tasks.md：任務清單（每個任務標註用什麼工具實作）
    ↓
analyze（必做）
    — 四維一致性檢查：Coverage / Consistency / Ambiguity / Gaps
    — 0 findings 才能繼續
    ↓
apply（執行）
    — 讀 tasks.md，依工具分配派工
    — 每個 Wave 完成後 commit
    ↓
ingest（需求變更時）
    — 更新 spec，重新分析
    ↓
archive（完成後）
    — 標記 change 完成，移入歸檔
```

## 為什麼「先寫 spec」讓開發更快

表面看起來：寫 spec 要花時間，是額外的工作。

實際上：

**Spec 把「思考的代價」提前到最便宜的時候付。**

| 什麼時候發現問題 | 修改代價 |
|------------|-------|
| Spec 階段（還沒寫 code）| 改幾行文字 |
| 開發中途 | 改代碼 + 改測試 + 可能要重構 |
| Code Review 後 | 重做那個功能 |
| 上線後 | 緊急修復 + 使用者已受影響 |

在 spec 階段發現「這個需求想清楚不對」，代價是 30 分鐘的討論。

在開發中途才發現，代價是兩天的返工。

## 為什麼「否決方案」要寫進 design.md

設計決策常見的形式：「我們選擇了 A。」

這樣的記錄沒有意義，因為六個月後看到這行，不知道當初為什麼排除 B 和 C。

正確的形式：

```
決策：選擇 A
原因：A 滿足 X 和 Y 的需求
否決 B：B 需要額外的 Z 依賴，我們想保持輕量
否決 C：C 只在 Tauri 環境可用，但我們的測試環境是瀏覽器
```

有了否決理由，六個月後如果 Z 的問題解決了，可以重新評估 B。
沒有否決理由，只知道「當初選了 A」，無法做出有根據的重新評估。

AIRE 的 mock backend 設計（D1）就有三個被否決方案：
- MSW：IPC 不是 HTTP，無法攔截
- 獨立 mock provider context：侵入性太高
- 環境變數開關：條件已足夠明確，不需要額外開關

這些否決理由讓任何接手的人都能快速理解設計邊界。

## SDD 的錯誤使用方式

### 錯誤 1：Spec 寫完就不改

Spec 是假說，假說在遇到新證據時應該更新。

「Spec 已經確定了，不能改」→ 這是把 SDD 當蓋房子在用。

正確做法：需求變更 → ingest 更新 spec → 重新 analyze → 繼續 apply。

### 錯誤 2：先做再補 spec

「先做出來看看，再補文件」→ 文件會永遠落後代碼。

落後的 spec 比沒有 spec 更糟糕，因為它會誤導後人。

### 錯誤 3：Spec 太細

Spec 不是代碼注釋，不應該寫到「第 42 行要做什麼」。

Spec 的粒度：功能層級的「做什麼」和「不做什麼」、接受標準、關鍵設計決策。

實作細節留給代碼。
