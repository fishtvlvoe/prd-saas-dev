# 熵減原則

這個檔案解決的問題：解釋「為什麼做少的比做多的更難，而且更對」，以及如何在實際開發決策中應用。

---

## 物理定義

熱力學第二定律：封閉系統的熵（混亂程度）只會增加，不會自動減少。

類比到產品開發：**不主動管理的系統，功能只會越加越多，複雜度只會越來越高，永遠不會自動變簡單。**

熵增是自然狀態，熵減是主動選擇。

---

## 商業上的熵增陷阱

### 功能堆積

| 階段 | 發生了什麼 |
|------|----------|
| 早期 | 核心功能上線，使用者 10 個 |
| 6 個月後 | 使用者反饋要 A、B、C 功能 |
| 1 年後 | 加了 A、B、C、D、E（競爭對手有 D、E） |
| 2 年後 | 有 20 個功能，使用者常用的只有 3 個 |
| 3 年後 | 每個版本都要維護 20 個功能，但新功能越來越難加 |

這是絕大多數 SaaS 的命運。不是因為做錯了什麼，是因為沒有主動做熵減。

### 需求膨脹

「使用者說他們想要 X」 ≠ 「X 應該進產品」

| 使用者說的 | 真正的需求 |
|----------|----------|
| 「能不能加合約管理？」 | 「我現在要切換好幾個工具，很麻煩」 |
| 「能不能有行事曆？」 | 「我想知道哪個案子快到期了」 |
| 「能不能整合 LINE？」 | 「我想快速通知客戶」 |

真正的需求往往比「使用者說的」更窄、更具體。解決真正的需求，不是解決「使用者說的」。

## 開發上的熵減實踐

### 什麼東西不加

**判斷流程**：

```
要加這個功能嗎？
  ↓
它解決的痛點是什麼？
  ↓
這個痛點是核心使用者真的有的嗎（有沒有驗證過）？
  ↓
加了這個功能，維護成本是多少？
  ↓
維護成本 < 解決的痛點價值 → 加
維護成本 ≥ 解決的痛點價值 → 不加
```

### 最小改動原則

修 bug 的時候，只改讓 bug 消失需要改的東西。

不「順手」重構附近的代碼（除非它直接造成 bug）。

不「順便」加「以後可能需要的」功能。

每個改動的範圍越小，回溯越容易，風險越低。

### 依賴管理

每加一個 npm 套件前問：
1. 這個功能我們自己能用 50 行以內實現嗎？（能的話自己寫）
2. 這個套件的維護狀態如何？（三年沒更新的套件不加）
3. 這個套件的體積對 bundle size 影響多少？

AIRE 案例：PDF 渲染用 `@react-pdf/renderer`，沒有自己造輪子。理由：PDF 渲染涉及字型、編碼、跨平台差異，50 行實現不了。依賴這個套件的代價遠小於自己做。

## AIRE 的熵減決策

AIRE 的功能邊界非常清楚：

**做的事**：
- 不動產說明書自動化產生
- 法規條款管理
- 品牌客製化（logo、公司名）
- 操作日誌

**刻意不做的事**：
- 客戶關係管理（CRM）
- 合約管理
- 行銷功能
- 客戶溝通（LINE 整合等）
- 財務管理

為什麼不做？因為核心使用者的核心痛點是「說明書製作太耗時」。其他功能解決的是不同的痛點，如果都做進 AIRE，AIRE 就從「說明書工具」變成「房仲 ERP」。

ERP 有它自己的市場，AIRE 不是那個產品。

## 開發流程的熵減

開發工作流中也有很多隱性的熵增：

| 熵增來源 | 症狀 | 熵減解法 |
|---------|------|---------|
| 每次測試都要打包 Tauri | 等 3 分鐘才能看結果 | Mock backend，pnpm dev 直接測 |
| 文件落後代碼 | 不知道哪個才對 | SDD，先寫 spec 再寫 code |
| Bug 修完不記錄 | 下次再踩同一個坑 | lessons.md，踩坑即時記錄 |
| 功能上線沒有測試 | 改 A 壞 B，不知道為什麼 | TDD，紅燈先跑通再實作 |

每個流程改進本質上都是「找到熵增的來源，主動消除它」。

---

## 對比表

| 熵增思維 | 熵減思維 |
|---------|---------|
| 加功能讓產品更強 | 只加真正需要的功能 |
| 使用者說要就加 | 先問使用者真正的需求是什麼 |
| 代碼越複雜說明越聰明 | 代碼越簡單越難被破壞 |
| 先做出來再說 | 先定義對的標準再動手 |
| 依賴多 = 功能強 | 依賴多 = 維護成本高 |
| 文件是事後的 | 文件是設計的一部分 |

---

## 實踐：熵減的三個問題

面對任何「要不要加」的決策，問這三個問題：

1. **它的職責是什麼？** 不能一句話說清楚 → 先不加。
2. **不加它會怎樣？** 沒有立即影響 → 先不加。
3. **加了之後，誰來維護它？** 沒有明確的維護責任 → 先不加。

三個問題都能回答，再考慮加。
