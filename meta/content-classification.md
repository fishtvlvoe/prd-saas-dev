# 內容分類規則（誰要讀這段？）

這個檔案解決的問題：每次新增知識前，先判斷「這段給誰看」，避免內容寫錯地方變成沒人讀的孤兒檔。

---

## 兩種讀者，兩種寫法

| 讀者 | 寫法 | 目的 |
|------|------|------|
| **人**（Fish、未來的自己、訪客） | 中文 + 哲學脈絡 + 表格 + 生活比喻 | 內化思維、回頭查為什麼這樣決定 |
| **AI agent**（Claude、Codex、GPT） | 英文 + step-by-step + 結構化指令 + 範例 | 直接照做、不需翻譯、不靠自由發揮 |

---

## 資料夾對應

```
給人看（中文）
├── site/          視覺化網頁，費曼學習法翻譯所有術語
├── why/           設計哲學、動機、為什麼這樣選
├── how/           工作流程、方法論、決策步驟
└── what/          具體產出、技術細節、踩坑紀錄

給 AI 看（英文）
└── ai-guide/      step-by-step 執行指南，AI agent 直接照做

中性（規則、索引）
├── README.md      入口、知識地圖
└── meta/          這份分類規則本身、貢獻指引
```

---

## 判別流程

新知識進來時，問三個問題：

```
Q1：這是「為什麼」、「方法論」、「決策依據」嗎？
  → 是 → 寫中文，放 why/ 或 how/
  → 否 → 進 Q2

Q2：這是「給 AI 照做的執行步驟」嗎？
  → 是 → 寫英文，放 ai-guide/
  → 否 → 進 Q3

Q3：這是「具體技術產出 / 模板 / 踩坑」嗎？
  → 是 → 寫中文，放 what/
  → 否 → 不知道放哪 → 問 Fish
```

---

## 雙寫情況（人 + AI 都要讀）

有些知識兩邊都需要，但寫法不同：

| 場景 | 中文版（人） | 英文版（AI） |
|------|------------|------------|
| Fish + Claude 協作流程 | how/client-discovery-workflow.md | ai-guide/client-discovery-workflow.md |
| 對應方法論 | how/client-opportunity-mapping.md | ai-guide/client-opportunity-mapping.md |
| SDD 工作流 | how/sdd-workflow.md | ai-guide/sdd-workflow.md |

**雙寫原則**：
- 中文版先寫，捕捉哲學跟脈絡
- 確認方法論穩定後再產英文版（避免改一邊忘另一邊）
- 英文版只保留「AI 要照做的部分」，砍掉哲學論述

---

## 反模式

- ❌ 把哲學脈絡寫進 ai-guide/（AI 不需要，浪費 token）
- ❌ 把英文 step-by-step 放 why/（人類讀起來累，看不到動機）
- ❌ 寫了中文就忘了同步英文（協作流程類最容易出現）
- ❌ 不確定放哪就硬塞到 README（README 是入口，不是垃圾桶）

## Fish 的習慣性指令對照

| Fish 說 | 我該做 |
|---------|--------|
| 「幫我放到資料夾」 | 判別人/AI/雙寫 → 放對位置 → 同步 README 索引 |
| 「這要給人看」 | 中文 why/how/what |
| 「這要給 AI 看」 | 英文 ai-guide/ |
| 「累積成資產」 | 默認進資料夾、commit、push（不只是貼在對話） |
| 「寫成 SOP」 | 給 AI 看 → ai-guide/；給人對焦 → how/ |
